Run Jetty in the Maven life-cycle

In the Java EE kingdom, JBoss Application Server is king. But sometimes it's nice to have a lighter container to run during our development phases. Here are some typical use cases:

  • quickly redeploy our fresh code or dependencies injections during development,
  • run web-based automated tests during the Maven life-cycle, in continuous integration basically.

Jetty is a good choice for these specific needs as it comes with multiple features:

  1. It can be started and stopped in any Maven life-cycle phases using a rich Maven plugin.
  2. It can run in the same Java process as Maven. No need to install any 3rd party software or run any external commands. This feature may be helpful in a continuous integration environment.
  3. It can run either a war package or scattered resources from multiple locations. The 2nd option is quiet useful during development, specially when dealing with war overlays.
  4. It can detect code changes, dependencies updates or any file modifications and then redeploy our application automatically.
  5. And, of course, it allows to debug code within Eclipse.

In this tutorial, I will show you how to set up a basic configuration of the Jetty Maven plugin, how to customize it for a developer use and how to use it for both server and client integration tests.

First configuration of the Jetty Maven plugin

First thing, we need to declare the Jetty Maven plugin in our pom.xml file.

NOTE: you can look for a more recent or specific version of the jetty-maven-plugin in the Maven Central Repository. The plugin version is pretty much the version of the embedded Jetty server. There are actually 3 branches:

  • version 9 (groupId: org.eclipse.jetty) is the latest version but it requires Java 7.
  • version 8 (groupId: org.mortbay.jetty) is the one we are going to use in this tutorial.
  • version 6 (groupId: org.mortbay.jetty, artifactId: maven-jetty-plugin) is outdated.
<project>
 ...
 <artifactId>my-artifact-id</artifactId>
 <packaging>war</packaging>
 ...
 <build>
  <plugins>
   <plugin>
    <groupId>org.mortbay.jetty</groupId>
    <artifactId>jetty-maven-plugin</artifactId>
    <version>8.1.9.v20130131</version>
    <configuration>
     <stopKey>STOP</stopKey>
     <stopPort>9999</stopPort>
     <scanIntervalSeconds>5</scanIntervalSeconds>
    </configuration>
   </plugin>
  </plugins>
 </build>
</project>

This is a very basic configuration:

  • stopKey and stopPort will be used to stop a running instance of Jetty using the goal jetty:stop.
  • scanIntervalSeconds is the period between each file scan for automatic redeployment. By default, this is disabled (value 0).

Let's start our web application in "scattered" mode using the goal jetty:run:

mvn jetty:run

We can run it in "packaged" mode as well using the goal jetty:run-war:

mvn jetty:run-war

In both cases, our web application will be accessible from the following URL:

http://localhost:8080/

Change the URL

For now, our application is served from the root location (/) of our web server. For many reasons, you may want to have the application accessible from /my-artifact-id/, for instance, in order to have the same conditions as your production environment. To do this, we need to add the following lines in the plugin configuration block:

    <configuration>
     ...
     <webAppConfig>
      <contextPath>/${project.artifactId}</contextPath>
     </webAppConfig>
    </configuration>

Once restarted, our web application will be accessible from the following URL:

http://localhost:8080/my-artifact-id/

Change the port

8080 is the default port for most application servers, but you will probably need to change it if you already have a running server on your machine or if you want to run multiple Jetty's instances in parallel. To do this, we just change the system property jetty.port on the command line:

mvn -Djetty.port=8181 jetty:run

Our web application will now be accessible from the following URL:

http://localhost:8181/my-artifact-id/

We could also have specified this in the project properties section of our pom.xml or, a bit trickier, have configured a specific Jetty connector, like that:

    <configuration>
     ...
     <connectors>
      <connector implementation="org.eclipse.jetty.server.nio.SelectChannelConnector">
       <port>8181</port>
      </connector>
     </connectors>
    </configuration>

Speed up the deployment

At this time, you are probably experiencing performance issues during deployment. If your web application hangs at start-up during more than a minute with a high CPU usage, you are surely dealing with the Servlet API 2.5 looking for annotations everywhere.

A workaround is to limit annotation search only to a few jars using a context configuration file.

Let's create a file named jetty-context.xml in the directory src/test/resources (so it won't be included in our web application) with the following content:

<?xml version="1.0" encoding="UTF-8"?>
<Configure class="org.eclipse.jetty.webapp.WebAppContext">
 <Call name="setAttribute">
  <Arg>org.eclipse.jetty.server.webapp.WebInfIncludeJarPattern</Arg>
  <Arg>^$</Arg>
 </Call>
</Configure>

Here we put an empty string as regular expression so we don't want the Servlet API to look for any annotations in any jar dependencies. But you can specify a more complex expression if needed.

Then, in our pom.xml file, we will add the following line in the plugin configuration block:

    <configuration>
     ...
     <contextXml>${project.build.testOutputDirectory}/jetty-context.xml</contextXml>
    </configuration>

jetty:run for development

The goal jetty:run suits well to a developer use. It allows:

  • Running our project without requiring that it is packaged. That's what I'm calling "scattered" mode because every component of the project is used from its origin location.
  • Modifying static resources (HTML, JavaScripts, Cascading Style Sheets...) and even dynamic resources that are translated at run-time (JSP, JSF...) without having to redeploy the application.
  • Redeploying automatically our web application when classes or resources change using the periodic file scan mechanism.

Serve multiple web resource directories

When running in "scattered" mode, resources can be located in many different directories because they are not yet packaged. Here are some typical scenarios:

  • Resources contained in war dependencies. Jetty handles war overlays even when using the goal jetty:run, but static resources can also be served from their origin directory to facilitate hot modifications.
  • Some test pages that are not located in the directory src/main/webapp because we don't want them in the assembled war file.
  • A Java applet that has to be retrieved from the repository using the maven-dependency-plugin.

By default when using the goal jetty:run, Jetty will serve web resources contained in the directory src/main/webapp. However, Jetty allows to have extra resource directories that can be served in addition to the default one. We will achieve this by adding a resourceBases block in the webAppConfig block:

     <webAppConfig>
      ...
      <resourceBases>
       <resourceBase>${project.basedir}/src/main/webapp</resourceBase>
       <resourceBase>${project.basedir}/../other-project/src/main/webapp</resourceBase>
       <resourceBase>${project.basedir}/src/test/webapp</resourceBase>
       <resourceBase>${project.build.directory}/java-applets</resourceBase>
      </resourceBases>
     </webAppConfig>

Custom web.xml

With Jetty, we can specify a different deployment descriptor from the default web.xml that is located in the directory src/main/webapp/WEB-INF. This way we can add or replace configuration (servlets, filters...) for development or test purposes.

One possibility is to write a completely different web.xml file that will replace the default one. This is possible by specifying a descriptor in the webAppConfig block:

     <webAppConfig>
      ...
      <descriptor>${project.basedir}/src/test/webapp/WEB-INF/devel-web.xml</descriptor>
     </webAppConfig>

A more convenient possibility is to override our web.xml by another file that contain only the additional or replacement configuration:

     <webAppConfig>
      ...
      <overrideDescriptor>${project.basedir}/src/test/webapp/WEB-INF/override-web.xml</overrideDescriptor>
     </webAppConfig>

In order to have full control of what servlets and filters need to be present or not, the most powerful solution would be to use XML entities. For instance, we could have:

  • a first entity containing declarations of all the shared servlets and filters required by both web.xml and devel-web.xml,
  • a second entity containing only declarations of the servlets and filters required by web.xml,
  • a third entity containing only declarations of the servlets and filters required by devel-web.xml.

In the default web.xml, we would have something like that:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE document [
<!ENTITY webEntity1 SYSTEM 'webEntity1.xml'>
<!ENTITY webEntity2 SYSTEM 'webEntity2.xml'>
]>
<web-app>
 &webEntity1;
 &webEntity2;
</web-app>

And the devel-web.xml would look like:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE document [
<!ENTITY webEntity1 SYSTEM '../../../main/webapp/WEB-INF/webEntity1.xml'>
<!ENTITY webEntity3 SYSTEM 'webEntity3.xml'>
]>
<web-app>
 &webEntity1;
 &webEntity3;
</web-app>

Extend the classpath

The Jetty Maven plugin sets up by default a duly classpath for our Java web application including our Maven dependencies, the classes and resources compiled from directories src/main/java and src/main/resources and the sub-directories WEB-INF/lib and WEB-INF/classes of our web resource directories if any (see above). But we are not bounded to that. There could be multiple reasons for extending the default Jetty's classpath, like using test classes or some custom dependencies injections in our developer environment for instance.

First, in order to set up a development or a test environment, we may want to include the classes and resources compiled from src/test/java and src/test/resources. There is actually a configuration flag named useTestScope for this:

    <configuration>
     ...
     <useTestScope>true</useTestScope>
    </configuration>

This will also include the Maven dependencies of scope test in the classpath. To include scope provided as well, just use the flag useProvidedScope. To add others specific Maven dependencies or to override the project dependencies, we can also add a dependencies block in the Jetty's plugin block:

   <plugin>
    ...
    <dependencies>
     <dependency>
      ...
     </dependency>
    </dependencies>
   </plugin>

Second, we may also want to add some specific directories or dependencies. To do so, we will add our extra classpath entries to the webAppConfig block of our plugin configuration:

     <webAppConfig>
      ...
      <extraClasspath>${project.build.directory}/custom-classes;${project.build.directory}/custom-dependency.jar</extraClasspath>
     </webAppConfig>

Will also be included all WEB-INF/lib and WEB-INF/classes sub-directories from our web resource directories that we configured previously. We can consider it as a third option of adding files in the classpath.

Extra scan locations

One of the biggest advantages of the Jetty Maven plugin is the periodic scan mechanism that can detect file modifications and redeploy the web application automatically. This feature, that we enabled earlier, is preconfigured to scan the project class and resource directories, as well as the Maven dependencies.

Because of the complex layout of our project, we probably will have to add some extra scan directories to the mechanism. One way is to add a scanTargets block in our plugin configuration:

    <configuration>
     ...
     <scanTargets>
      <scanTarget>${project.basedir}/src/main/webapp/WEB-INF/classes</scanTarget>
      <scanTarget>${project.basedir}/src/test/webapp/WEB-INF/classes</scanTarget>
     </scanTargets>
    </configuration>

For even more complexity, there is an other way which allows to specify patterns:

    <configuration>
     ...
     <scanTargetPatterns>
      <scanTargetPattern>
       <directory>${project.basedir}/..</directory>
       <includes>
        <include>other-project1/target/*.jar</include>
        <include>other-project2/src/**/*.xml</include>
       </includes>
      </scanTargetPattern>
     </scanTargetPatterns>
    </configuration>

jetty:run-war for tests

The goal jetty:run-war is definitely not suitable for development since it's bound to the Maven phase package and requires an assembled war. However it's appropriate for people who want to deploy a freshly built war package and perform some automated tests in the Maven life-cycle. Combined with the maven-failsafe-plugin it's a powerful solution for web oriented integration tests.

Here is the execution blocks we need to add in the jetty-maven-plugin section of our pom.xml:

   <plugin>
    ...
    <executions>
     <execution>
      <id>start-jetty</id>
      <phase>pre-integration-test</phase>
      <goals>
       <goal>stop</goal>
       <goal>run-war</goal>
      </goals>
      <configuration>
       <scanIntervalSeconds>0</scanIntervalSeconds>
       <daemon>true</daemon>
      </configuration>
     </execution>
     <execution>
      <id>stop-jetty</id>
      <phase>post-integration-test</phase>
      <goals>
       <goal>stop</goal>
      </goals>
     </execution>
    </executions>
   </plugin>

First execution block, which is bound to the pre-integration-test phase, is used to start the Jetty server before the integration tests. The goal jetty:stop is invoked to stop any previous instance of the server (using stopKey and stopPort values) in case it's still running. The run-war goal is invoked to start the server and to deploy the packaged war file. <scanIntervalSeconds> set to 0 disables hot deployment scanning. <daemon> set to true tells the server to continue running after the end of the goal.

Second execution block, which is bound to the post-integration-test phase, is used to stop the Jetty server after the integration tests using the goal jetty:stop.

jetty:deploy-war for client tests

Another interesting goal, jetty:deploy-war can deploy any war file without having to assemble it. This can be useful in a web client project with automated integration tests. We can easily imagine a scenario where the war package of the server project is retrieved by the client project using the maven-dependency-plugin and automatically deployed to run some client tests.

Here is a configuration sample for our client project:

   <plugin>
    ...
    <configuration>
     ...
     <war>${project.build.directory}/copied-from-server-project.war</war>
    </configuration>
    <executions>
     <execution>
      <id>start-jetty</id>
      <phase>pre-integration-test</phase>
      <goals>
       <goal>stop</goal>
       <goal>deploy-war</goal>
      </goals>
      <configuration>
       <scanIntervalSeconds>0</scanIntervalSeconds>
       <daemon>true</daemon>
      </configuration>
     </execution>
     <execution>
      <id>stop-jetty</id>
      <phase>post-integration-test</phase>
      <goals>
       <goal>stop</goal>
      </goals>
     </execution>
    </executions>
   </plugin>

Conclusion

These configuration examples are actually a small illustration of the Jetty Maven plugin potential. Jetty can also be configured with external XML files which extend widely the possibilities. You will find more information about Jetty and the Jetty Maven plugin on the Jetty's website.

However, there are still a few bad points regarding the use of Jetty and its plugin:

  • The documentation is fragmented. Version 6 documentation is hosted at jetty.codehaus.org. Newer versions are hosted by eclipse.org but version 7 and 8 are at wiki.eclipse.org and version 9 is at www.eclipse.org/jetty. The 3 branches aren't fully documented and each branch's documentation may contain relevant information.
  • The automated redeployment mechanism implemented by the plugin is a bit old-fashioned compared to class reloading solutions provided by JRebel or DCEVM. There is no doubt that these technologies will be part of Java in the future.
